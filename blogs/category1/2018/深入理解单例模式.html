    确实是这样没错了，但是你总得让人家去获得 你的一个对象的方法呀。所以我们的做法是：在Dog类中自己去new一个对象，然后设置一个外界可以获取Dog对象的静态方法即可。

    public class Dog {
    private static Dog dog = new Dog();
    private Dog() {}
    public static Dog getInstance() {
        return dog;
    }
}

    一个最基础的单例模式就出来了！！这就是饿汉式单例模式（所谓饿汉式就是 对象伴随着类加载出来，不管是否用得到！）；

    那问题来了：如果外界用不到该对象的属性的东西，属性也跟着类的加载而加载，这样很占用内存，有什么办法可以改进这个呢？那么懒汉式单例模式就出现了。

    public class Dog {
    private static Dog dog;
    private Dog() {}
    
    public static Dog getInstance() {
        if(dog == null) {
            dog =  new Dog();
        }
        return dog;
    }
}

    由此我们可以看出，并发情况下，产生的并不是单例对象，所以还是需要改进单例模式。无非就是加锁嘛，直接点！！

    public void Dog {
    private static Dog dog;
    private Dog() {}
    public static Dog getInstance() {
        if (dog == null) {
            synchronized (Dog.class) {
                if (dog == null) {
                    dog = new Dog();
                }
            }
        }
        return dog;
    }
}

    然后在测试，全部的对象的地址都是同一个了，但是还是有问题呀，因为中间的new Dog()不是原子性操作，我就在代码里讲述：

    public static Dog getInstance() {
        if (dog == null) {
            synchronized (Dog.class) {
                if (dog == null) {
                    dog = new Dog();
                    /*
                    1. 分配内存
                    2. 执行构造方法，初始化对象
                    3. 把这个对象指向这个空间
                    那么在多线程的情况下，有可能发生指令重排，即执行步骤是132而不是123，那么下一个线程获取的时候dog不是被初始化的对象，就出问题了！！！
                    */
                }
            }
        }
        return dog;
    }

    所以，为了避免指令重排的情况，Dog dog属性需要加一个关键字：volatile

    // 完整版
public void Dog {
    private volatile static Dog dog;
    private Dog() {}
    public static Dog getInstance() {
        if (dog == null) {
            synchronized (Dog.class) {
                if (dog == null) {
                    dog = new Dog();
                }
            }
        }
        return dog;
    }
}

    该单例模式就真的无懈可击了嘛？不是的，假如我用反射！就可以获得类的私有的方法和属性了呀。

    public class Test {
    public static void main(Sting[] args) {
        Dog dog = Dog.getInstance();
        Constructor<Dog> dogConstructor = Dog.class.getDeclaredConstructor(null);
        dogConstructor.setAccessible(true);
        Dog dog1 = dogConstructor.newInstance(null);
        System.out.println(dog);
        System.out.println(dog1);
    }
}

    很明显，反射已经破坏了单例模式。

    那么我Dog类如何改进呢？因为反射是从无参构造方法中获取到类的私有属性的，那么我们可以在构造方法中可以增加一个判断来不让反射获取到构造方法。

    public void Dog {
    private volatile static Dog dog;
    private Dog() {
        synchronized(Dog.class) {
            if(dog != null) {
                throw new RuntimeException("不要使用反射试图破坏单例模式");
            }
        }
    }
    public static Dog getInstance() {
        if (dog == null) {
            synchronized (Dog.class) {
                if (dog == null) {
                    dog = new Dog();
                }
            }
        }
        return dog;
    }
}

    所以出来了枚举类的单例模式，是可以真正防止反射进行破坏的。

    public enum Dog {
    INSTANCE;

    public Dog getInstance() {
        return INSTANCE;
    }
}
class Test {
        public static void main(String[] args) {
        Dog dog = Dog.INSTANCE;
        Dog dog1 = Dog.INSTANCE;
        System.out.println(dog.hashCode());
        System.out.println(dog1.hashCode());
    }
}
