    OS 之网络模型

    考小拉 9/10/2023 OS秋招

    以下内容参考 小林coding (opens new window) 并做总结

    # 什么是 DMA 技术？

    因为我们的零拷贝技术是基于这个 DMA 技术的，所以我们需要知道什么是 DMA 技术，所谓的 DMA 技术就是内存直接访问技术，简单来说，在我们进行IO设备和内存的数据传输的时候，数据搬运工作不需要 CPU 来执行，而是交给我们的 DMA 控制器来执行，而 CPU 就不需要参与任何数据搬运的事情，只需要负责发指令（传输什么数据，从哪里传输到哪里等）即可，这样 CPU 就能执行其他的事务了。

    # 传统传输文件的过程

    举个例子：我们进行一次读写文件；

    可以看出发生了四次的用户态到内核态的上下文切换，因为发生了 read() 和 write() 嘛，每次系统调用都需要从用户态切换到内核态，等内核完成之后再从内核态转换为用户态。

    期间还发生了四次数据的拷贝，其中两次是（磁盘控制器 ---> 内核缓冲区 || socket缓冲区 ---> 网卡缓冲区） DMA 拷贝，两次是（内核缓冲区 ---> 用户缓冲区 || 用户缓冲区 ---> socket 缓冲区）内核缓冲区即 CPU拷贝；

    本想着搬运一次的数据，却需要搬运四次，所以说，必须要减少这个上下文切换的次数以及这个数据拷贝的次数。

    我们可以从图中看到，从 【内核缓冲区 拷贝到 用户缓冲区】以及【用户缓冲区 拷贝到 socket 缓冲区】这个过程是没必要的，因为传输数据过程中不会对数据进行加工。

    # 实现零拷贝技术

    1. mmap + write（了解）
    2. sendfile

    这里重点讲一下这个 sendfile 技术，我们现在的 linux 也支持这个 SG-DMA 技术（这一步可以减少 内核缓冲区 ---> socket 缓冲区 这个过程，直接从内核缓冲区 ---> 网卡缓冲区）；内核缓冲区只需要将 这个描述符和数据长度 发送给 socket 缓冲区

    这就是所谓的零拷贝技术了，我们没有在内存层面去拷贝技术，也就是说全程没有通过这个 CPU 来搬运数据，全部都是由这个 DMA 技术来进行搬运的。传统的文件传输需要四次的上下文切换以及四次的数据拷贝，而我们的零拷贝只需要两次的上下文切换以及两次的数据拷贝即可，而且我们的数据拷贝都不需要 CPU 来参与，都是我们的 DMA 控制器来进行参与的；

    我们提到的内核缓冲区是啥？

    其实就是 pageCache，两个主要的特点：第一个是 可以缓存最近被访问的数据（局部性原理）；第二点是有这个预读的功能，当 read() 的时候虽然只是读到了 0- 32 字节的数据，但是可以把后面的 32- 64 字节的数据也读取到pageCache 中，如果在 32～64 KB 淘汰出 PageCache 前，进程读取到它了，收益就非常大。

    大文件的话就不要用这个零拷贝技术了，主要原因是 内存的空间小，如果一个大文件拷贝到内存中，我们的热点小文件就没办法被读到了；大文件可以采取 异步IO 的方式进行；

    # IO多路复用

    一个进程来维护多个 Socket

    我们熟悉的 select/poll/epoll 内核提供给用户态的多路复用系统调用，进程可以通过一个系统调用函数从内核中获取多个事件。这三个技术是如何获取网络事件的呢？在获取事件时，先把所有连接（文件描述符）传给内核，再由内核返回产生了事件的连接，然后在用户态中再处理这些连接对应的请求即可。

    select 实现多路复用的方式：将已连接的 Socket 都放在一个文件描述符集合（BitsMap 大小为1024）中，然后调用 select 函数将文件描述符集合拷贝到内核里，让内核检查是否有网络事件的发生，通过遍历的方式进行检查的，将有事件产生的进行标记为可读或者可写，然后将这个文件描述符集合从内核拷贝回用户态，用户态再进行遍历，然后进行处理。poll 其实也是差不多，只是换了数据结构，变成了一个动态数组，以链表的形式组织；

    epoll 解决了上面的 select 以及 poll 的问题，第一点：采用了红黑树来跟踪进程所有待检测的文件描述字，把需要监控的 socket 加入到内核的红黑树中，因为 select 以及 poll 没有这种待检测的 socket 的数据结构，所以每次操作都需要传入整个 socket 集合给内核，而 epoll 有这种待检测的红黑树结构，所需要只需要传入一个 待检测的 socket 即可；第二点：epoll 采用事件驱动机制，内核里维护了一个链表来记录就绪事件，当某个 socket 有事件发生的时候，通过回调函数内核会将其加入到这个就绪事件列表中，最后用户会返回有事件发生的文件描述符个数即可，不需要扫描整个文件描述符集合；

    这里也了解一下这个事件触发模式：边缘触发以及这个水平触发；

    边缘触发就是只有第一次满足事件的时候触发，之后都不会进行传递同样的事件了；水平触发的就是只要满足事件的条件，比如内核中有数据需要读，就一直不断地把这个事件传递给用户；

    # 高性能网络模型

    • Reactor 负责监听和分发事件，事件类型包括连接事件、读写事件；
    • 处理资源池负责处理事件，如 read -> 业务逻辑 -> write

    三种方案：

    • 单 Reactor 单进程 / 线程；
    • 单 Reactor 多线程 / 进程；
    • 多 Reactor 多进程 / 线程；

    第一个：单 Reactor 单进程

    • Reactor 对象作用是监听和分发事件；
    • Acceptor 负责建立连接；
    • Handler 负责处理业务

    两个缺点：①没用利用多核 CPU 这个性能；②Handler 对象业务处理的时候，整个进程是没有办法处理其他连接事件的，如果业务处理耗时长，响应会延迟。

    单 Reactor 单进程的方案不适用计算机密集型的场景，只适用于业务处理非常快速的场景。比如说：Redis

    第二个：单 Reactor 多进程

    这里和 单 Reactor 单进程 有一个区别：就是 Handler 不在处理业务，而是只是负责数据的接受和发送，将任务都交给了线程池的 Processor 进行处理，处理完后，将结果发送给主线程的 Handler 对象，接着由 Handler 通过 send 方法将结果发给 client。

    「单 Reactor」的模式还有个问题，因为一个 Reactor 对象承担所有事件的监听和响应，而且只在主线程中运行，在面对瞬间高并发的场景时，容易成为性能的瓶颈的地方

    第三个：多 Reactor 多进程

    方案详细说明如下 ：

    • 主线程中的 MainReactor 对象通过 select 监控连接建立事件，收到事件后通过 Acceptor 对象中的 accept 获取连接，将新的连接分配给某个子线程；
    • 子线程中的 SubReactor 对象将 MainReactor 对象分配的连接加入 select 继续进行监听，并创建一个 Handler 用于处理连接的响应事件。
    • 如果有新的事件发生时，SubReactor 对象会调用当前连接对应的 Handler 对象来进行响应。
    • Handler 对象通过 read -> 业务处理 -> send 的流程来完成完整的业务流程。

    实际的项目就是有：Netty、Memcache

    Reactor 是非阻塞的同步网络模式，而 Proactor 是异步网络模式。

    • 阻塞：典型的就是我们阻塞 BIO 了，当我们进行 read() 时，线程会被阻塞，直到我们内核数据准备好了，并把内核数据拷贝到我们的应用程序缓冲区，拷贝完成后，这个 read() 才进行返回。

      注意：阻塞等待就是【内核数据准备好】和【数据从内核缓冲区拷贝到应用程序缓冲区】这两个过程

    • 非阻塞：典型就是我们的 NIO 了，非阻塞的 read() 请求在数据未准备好的情况下就立即返回，可以继续往下执行，此时应用程序不断轮询这个内核，直到数据准备好，内核将数据拷贝到应用程序缓冲区，read() 调用才获取到结果。

      但是，注意：这里最后一次 read() 调用，获取数据的过程是一个同步的过程，需要进行等待，这里的同步指的是内核态的数据拷贝到用户程序的缓冲区这个过程。

      因此，无论是 read 还是 send 是阻塞 IO，还是非阻塞IO 都是同步调用的。因为在 read 调用时，内核将数据从内核空间拷贝到用户空间这个过程都是需要进行等待的，也就是这个过程是同步的；

    **而真正的异步 IO 是【数据在内核准备好】和【数据从内核空间拷贝到用户应用程序缓冲区】两个空间都是不需要等待的； ** 应用程序不需要自己主动发出拷贝数据的动作。

    Reactor 以及 Proactor 区别：

    • Reactor 是非阻塞同步网络模式，感知的是就绪可读写事件；即在每次感知有事件发生后，就需要应用程序主动调用 read 方法进行完成数据的读取，也就是需要应用进程主动将 socket 接受缓存中的数据读到应用程序中的内存来，这个过程是同步的，读取完数据后才能进行处理；
    • Proactor 是非阻塞异步网络模式，感知的是已完成的读写事件；总的来说，在发起异步读写请求后，就是统统不用干，等 OS 来通知我们应用程序直接处理数据。
    • 其实两者都是基于【事件分发】的网络编程模式。
