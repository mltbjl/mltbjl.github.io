    # 内存管理

    • 运行时区域：一些概念还是要了解的

      程序计数器：pc 寄存器，其实也是一块缓存，就是存储当前线程执行到哪一个步骤

      Java 虚拟机栈：方法调用的时候，会创建栈帧，用来存储局部变量表，引用变量，操作数栈等 Java 中的普通方法的调用都是通过栈来实现的；方法调用的数据需要通过栈进行传递，每一次方法调用都会有一个对应的栈帧被压入栈中，每一个方法结束后，就会有一个栈帧被弹出。

      堆：

      方法区：存储虚拟机已经加载的类的类型信息、常量、静态变量等

    • 对象创建过程

      图片 (opens new window)

      首先检查类是否被加载过 --- 然后给对象分配内存 --- 然后初始化对象设置为 0 值（除了对象头） --- 给对象头进行初始化 --- 执行 init() 方法

    • JVM 里 new 对象时，堆会发生抢占吗？JVM 是怎么设计来保证线程安全的？

      会发生堆抢占

      图片 (opens new window)

      1. 通过 CAS 分配重试的方式来保证这个操作的原子性
      2. 每个线程在 Java 堆中预先分配一小块内存，也就是本地线程分配缓冲（TLAB），要分配内存的线程，先在本地缓冲区进行分配，只有本地缓冲区用完了，分配新的缓冲区才需要同步锁定；

    • 内存泄漏和内存溢出

      内存泄漏：【占着茅坑不拉屎】，就是该对象已经不被使用了，但是没有被垃圾回收器清除

      内存溢出：内存大小是有限的，当我们的内存达到限度之后就会出现这个问题；【堆内存溢出和栈内存溢出】

      • 堆内存溢出：只要不断创建不可被回收的对象，比如静态变量，随着对象数量增多，就会超出堆内存产生堆内存溢出。

      直接计数；

      GC Root 有哪些：四种对象①方法区中类静态属性引用的对象；②方法区中常量引用的对象；③虚拟机栈（栈帧中本地变量表）中引用的对象；④调用了本地方法（Native）的对象

    • 对象引用有哪几种？

      强弱软虚

      这里了解一下虚引用就好：虚引用不会得到我们的对象实例，为一个对象设置虚引用关联的唯一目的是为了能在这个对象被收集器回收时收到一个系统通知。跟踪对象被垃圾回收的活动。

    • Java 堆的内存分区

      图片 (opens new window) 为什么要有 Survivor 区？有 Eden 区不就够了吗？

      1. 老年代对象的回收：Survivor 区中的对象经过多次垃圾回收仍然存活的对象会被晋升到老年区；如果没有 Survivor 区，那么所有的对象都只能在 Eden 区晋升在老年区，会导致老年代的对象增加，触发老年代的垃圾回收很消耗性能；
      2. 分代垃圾回收算法：Survivor 区是分代垃圾回收算法的一部分，通过将新生代分为 Eden 区和 Survivor 区，可以根据对象的存活情况在两个 Survivor 区之间进行对象的复制和清理，实现高效地垃圾回收。

    • 垃圾回收算法

      标记 --- 清除

      图片 (opens new window)

      缺点：主要是 ①内存碎片化的问题；②如果 Java 堆中含大量对象，而其中大部分需要回收的，这时必须进行大量标记和清除的动作，效率很低；

      标记 --- 复制

      图片 (opens new window)

      该算法解决了 垃圾清除算法面对大量的可回收对象时执行效率的问题；

      缺点：空间利用率不大，每次都是利用一半。

      新生代就是用这个算法的，因为新生代存活少，每次只需要复制少量的对象即可；

      标记 --- 整理

      图片 (opens new window)

      主要用于老年代，因为对象移动是很需要 进行 STW 操作，从吞吐量来考虑，老年代就需要使用 标记 --- 整理算法更合适。

    • Minor GC(Young GC)、Major(Old GC)、Mixed GC、Full GC

      • 部分回收

        Minor GC：新生代的垃圾回收

        Major GC：老年代的垃圾回收

      • 整集回收

        Full GC： 收集整个 Java 堆和方法区的垃圾回收

      Young GC触发的时机是：我们的 Eden 区没有足够的空间的时候就需要出发这个 Young GC 回收了。

      Full GC 触发时机：总的来说，都是因为各种预操作导致老年代空间不足，或者说是老年代空间就是不足导致的 或者是 System.gc() jmap - dump 等命令会触发 full gc

      图片 (opens new window)

    • 对象什么时候进入老年区

      图片 (opens new window)

      GC很多次都还存活的对象；大对象直接进入老年区（数组、长字符串）；空间分配担保（假如 Young GC 之后，新生代还有大量对象存活就需要老年代进行空间分配担保，把 Survivor 无法容纳得对象直接送入老年区）

    • 什么是 STW?什么是安全点？

      进行垃圾回收过程中，会涉及到对象的移动；为了保证对象引用更新的正确性，必须要暂停用户线程。像这样的停顿就是所谓的 STW ；

      那我们暂停线程是在任意时候都可以暂停线程吗？不是的，我们只能在一个叫做【安全点】的位置进行暂停线程，这些安全点的位置可以在：①循环的末尾；②方法返回前，或者是调用方法的 call 指令后；③可能抛出异常的位置。

    • 垃圾回收器

      图片 (opens new window)

      这里需要重点了解 CMS 以及 G1，还有了解一下我们虚拟机中默认的 Parallel Scavenge 和 Parallel Old

      先从 Parallel Scavenge 说起，这个收集器是新生代收集器，基于 标记 --- 复制 算法实现的；

      然后 Parallel Old 说起，这个收集器是老年代收集器，基于 标记 --- 整理 算法实现的；

      CMS 也是一款老年代收集器，基于 标记 --- 清除 算法实现的；它是一种以获取最短回收停顿时间为目标的收集器。

      G1 是以局部收集的思想和基于 Region 的内存布局形式；

    • 详细唠嗑一下 CMS 和 G1

      CMS 的工作流程：（Concurrent Mark Sweep）

      图片 (opens new window)

      • 初始标记：单线程执行，需要 STW，标记 GC Roots 能直达的对象；
      • 并发标记：无停顿，和用户线程同时进行，从 GC Roots 直达对象开始遍历整个对象图，这个阶段结束后，不一定记录了所有的可达对象，因为用户线程会更新节点的引用，所以无法保证可达性分析的实时性，所以这个步骤还会标记更新节点引用的地方；
      • 重新标记：多线程运行，需要 STW，并发标记阶段产生对象
      • 并发清除：无停顿，和用户线程同时运行，清除掉标记阶段标记的死亡的对象

      G1：（Garbage First）

      特点：G1 把连续的 Java 堆划分为多个大小相等的独立区域（Region），每个 Region 都可以根据需要，扮演新生代的 Eden 空间、Survivor 空间、或者老年代空间。

      这样就避免了回收整个堆，而是按照若干个 Region 集进行收集，同时维护一个优先列表，跟踪各个 Region 回收的价值，优先回收价值高的 Region

      回收过程跟我们的 CMS 差不多 图片 (opens new window)

    • 有了 CMS，为什么还需要 G1

      CMS 优缺点：第一个是这个算法会导致比较多的内存碎片；第二个就是在并发清除阶段用户线程依旧执行，有可能产生的浮动垃圾，本次垃圾清除无法清理，需要等到下一次的垃圾回收才能进行回收；

      G1 主要是解决了 CMS 的内存碎片的问题。

    # JVM 调优了解

    • 线上服务 CPU 占用过高如何排查？

      问题分析：CPU 占用过高的原因就是因为某个应用程序长期占用了 CPU 的资源

      图片 (opens new window)

      1. 先找出哪个进程占用 CPU 高

        TOP 命令查看系统各个进程的资源占用情况

      2. 然后根据找到对应进行进程里哪个线程占用 CPU 高

        TOP -Hp 进程ID 列出对应进程里的线程占用资源的情况

      3. 找到对应线程 ID 后，再打印对应线程的堆栈信息

        printf "%x\n" PID 把线程ID转换为16进制

        jstack PID 打印出进程的所有线程信息，从打印出来的线程信息中找到上一步转换为16进制的线程ID对应的线程信息。

      4. 最后根据线程的堆栈信息定位到具体的业务逻辑代码找到问题所在。

    • 内存飙高的问题如何排查？

      问题分析：内存飙高如果是发生在Java进程上，一般是因为大量对象被创建导致的，持续飙高的原因是垃圾回收跟不上这个对象创建速度，或者内存泄漏导致对象无法回收

      1. 观察垃圾回收的情况

        查看 GC 次数

        查看堆内存中占用空间最大的前几十个对象

        如果每次 GC 次数正常，而且回收的内存空间也正常，则就是因为对象创建速度快导致内存一致占用很高；如果每次回收的内存少，那么很可能因为内存泄漏而导致内存一致没有被回收。

    # 虚拟机执行

    • 类的生命周期

      加载类的字节码文件 --- 连接（验证、准备、解析）--- 初始化 --- 使用 --- 卸载

    • 类加载过程

      图片 (opens new window)

      三部分：

      1. 通过这个类的全限类名来获取定义此类的二进制字节流；
      2. 通过该字节流所代表的静态存储结构转换为方法区中运行时的数据结构；
      3. 在内存中生成该类的 java.lang.Class 对象，作为方法区这个类的各种数据的访问入口。

    • 类加载器

      四个：

      启动类加载器（BootStrap ClassLoader）：加载 Java 的核心库

      扩展类加载器（Extension ClassLoader）：加载 Java 的扩展库

      应用程序类加载器（Application ClassLoader）：根据 Java 应用的类路径（ClassPath）加载类，可以通过 ClassLoader.getSystemClassLoader() 来获取

      自定义类加载器

    • 什么是双亲委派机制？

      图片 (opens new window)

      当我们进行加载一个类的时候，我们的类加载器不会主动尝试进行类加载，而是让其父类加载器进行加载，如果父类加载器不能加载，我们的子类加载器才会对这个类进行加载。

    • 为什么要用双亲委派机制？

      保证类不被重复加载 以及 保证程序稳定运行

      例如类 java.lang.Object，它存在 rt.jar 下，通过双亲委派机制，保证最终都是委派给处于模型最高端的启动类加载器进行加载，保证了 Object 一致；

      反之，如果都由各个类加载器自行加载的话，如果用户自己也编写了一个名为 java.lang.Object 的类，并放在 ClassPath 中，那系统就出现不同的 Object 了。

    • 如何破坏双亲委派机制？（不能说破坏）

      双亲委派中默认有三个类加载器，最顶层的 BootStrapClassLoader 启动类加载器只负责加载 Javahome lib 下面的 jar 包，虚拟机决定的，想要打破就要重写 classLoader() 方法，修改这个类加载器的【活动范围】；但是真正的打破双亲委派的意义在于：如果是一个第三方的或者不在 classPath 下面的任意 jar 包，如何加载？只有你需要去加载一个不按套路出牌或者说你运行时需要新输入的类，这才需要打破双亲委派。

      延申出来一个问题：Tomcat 打破双亲委派了嘛？没有，因为 Tomcat 是一个 web 容器，需要用部署很多应用程序，所以就可以带来不同程序依赖第三方库的不同版本的问题，但是不同版本的类库中某一个类的全路径名可能是一样的，比如说：如多个应用都要依赖 hollis.jar，但是A应用需要依赖1.0.0版本，但是B应用需要依赖1.0.1版本。这两个版本中都有一个类是 com.hollis.Test.class。如果采用默认的双亲委派类加载机制，那么无法加载多个相同的类。Tomcat 还是依旧按照我们的类加载器进行启动的，只不过自己延申了自定义的类加载器 Common 类加载 & Shared 类加载器 & Catalina & 类加载器 & WebApp 类加载器。
