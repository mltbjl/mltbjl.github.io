    # 自定义线程池时线程池参数如何设置？

    场景：在程序进行过程中，不能保证一次计算出合适的参数，那么我们是否可以将修改线程池的参数的成本降下来？这样至少可以发生故障的时候可以快速调整从而缩短故障恢复的时间呢？

    基于这个思考，我们是否可以将线程池的参数从代码中迁移到分布式配置中心上，实现线程池参数可动态配置和即时生效？

    # 现有的解决方案的痛点

    我们线程池中的任务分为 CPU 密集型 以及 IO 密集型

    如果是 CPU 密集型，则核心线程数设置为 核心数 + 1，为什么是 + 1 呢？可以理解为一个备份的线程即可；（《Java并发编程实战》原话：即使当计算（CPU）密集型的线程偶尔由于页缺失故障或者其他原因而暂停时，这个“额外”的线程也能确保 CPU 的时钟周期不会被浪费。）

    但是这里如果你的服务器部署不止一个应用，就得考虑其他应用的线程池配置情况，其他的应用有可能抢 CPU ；

    如果是 IO 密集型，CPU * 2，就是把所有的任务都当作 IO 密集型去处理了。而且一个线程中一般来说不止一个自定义的线程池，比如有专门处理数据上送的线程池，有专门处理查询请求的线程池，这样去做一个简单的线程隔离。但是如果都用这样的参数配置的话，是不合理的；

    这里多一个疑问？专为面试而用：为什么核心线程数是 CPU * 2 呢？

    1. IO 密集型任务通常涉及文件读写、网络请求等操作，这些操作的执行时间相对于 CPU 执行任务来说往往较长。在这种情况下，线程会频繁地进行 IO 操作，而不是在 CPU 上执行计算密集型任务。将 IO密集型任务的核心线程数设置为 CPU * 2 可以确保在执行 IO 操作时，有足够的线程可用。这是因为在一个线程进行 IO 操作的同时，其他线程可以并行地执行其他任务，从而提高整体系统的吞吐量和性能。
    2. 同时，设置核心线程数为 CPU * 2 还可以避免线程饥饿的情况发生。线程饥饿是指线程在等待资源的情况下无法被调度执行，导致系统性能下降。通过设置足够多的核心线程数，可以减少线程饥饿的概率，提高系统的响应能力。

    # 动态更新的工作原理是啥？

      public final class NamedThreadFactory implements ThreadFactory {

    private final AtomicInteger threadNum = new AtomicInteger();
    private final String name;

    /**
     * 创建一个带名字的线程池生产工厂
     */
    public NamedThreadFactory(String name) {
        this.name = name;
    }

    @Override
    public Thread newThread(Runnable r) {
        Thread t = new Thread(r);
        t.setName(name + "[#" + threadNum.incrementAndGet() + "]");
        return t;
    }
}

    • 示例

      public class ThreadPoolDemo {
    public static void main(String[] args) throws InterruptedException {
        dynamicModifyExecutor();
    }

    /**
     * 自定义线程池
     */
    public static ThreadPoolExecutor buildThreadPoolExecutor() {
        return new ThreadPoolExecutor(2,
                5,
                60,
                TimeUnit.SECONDS,
                new ResizeLinkedBlockingQueue<>(10),
                new NamedThreadFactory("刘东生线程池"));
    }

    /**
     * 先提交任务给线程池，并且修改线程池参数
     */
    private static void dynamicModifyExecutor() throws InterruptedException {
        ThreadPoolExecutor executor = buildThreadPoolExecutor();
        for (int i = 0; i < 15; i++) {
            executor.submit(() -> {
                threadPoolStatus(executor, "创建任务");
                try {
                    TimeUnit.SECONDS.sleep(10);
                } catch (InterruptedException e) {
                    e.printStackTrace();
                }
            });
        }
        threadPoolStatus(executor, "改变之前");
        TimeUnit.SECONDS.sleep(2);
        executor.setCorePoolSize(10);
        executor.setMaximumPoolSize(10);
        executor.prestartCoreThread();
        ResizeLinkedBlockingQueue queue = (ResizeLinkedBlockingQueue) executor.getQueue();
        queue.setCapacity(100);
        threadPoolStatus(executor, "改变之后");
        Thread.currentThread().join();
    }

    /**
     * 打印线程池状态
     */
    private static void threadPoolStatus(ThreadPoolExecutor executor, String name) {
        ResizeLinkedBlockingQueue queue = (ResizeLinkedBlockingQueue) executor.getQueue();
        System.out.println(new SimpleDateFormat("dd-MM-yyyy HH:mm:ss").format(new Date()) + Thread.currentThread().getName() + "-" + name + "-：" +
                "核心线程数" + executor.getCorePoolSize() +
                "活动线程数" + executor.getActiveCount() +
                "最大线程数" + executor.getMaximumPoolSize() +
                "线程池活跃度" + divide(executor.getActiveCount(), executor.getMaximumPoolSize()) +
                "任务完成数" + executor.getCompletedTaskCount() +
                "队列大小" + (queue.size() + queue.remainingCapacity()) +
                "当前排队线程数" + queue.size() +
                "队列剩余大小" + queue.remainingCapacity() +
                "队列使用度" + divide(queue.size(), queue.size() + queue.remainingCapacity()));
    }

    private static String divide(int activeCount, int maximumPoolSize) {
        return String.format("%1.2f%%",
                Double.parseDouble(activeCount + "") / Double.parseDouble(maximumPoolSize + "") * 100);
    }
}

    效果是实现了，但是原理是啥呢？

    setCorePoolSize() 方法：美团的文章提出：当然这也是 setCorePoolSize() 的一个解释 setCorePoolSize() 流程图

    ” 在运行期间线程池调用此方法设置 corePoolSize 后，线程池会直接覆盖原来的 corePoolSize 值，并且基于当前值和原始值比较结果采取不同的处理策略！“

    ” 对于当前值小于当前工作线程数的情况，说明有多余的 worker 线程，此时会向当前 idle 的 worker 线程发起中断请求以实现回收，多余的 worker 在下次 idle 的时候也会进行被回收 “

    ” 对于当前值大于原始值且当前队列有待执行任务的情况，则线程池会创建新的 worker 线程来执行队列任务 “

    setMaximumPoolSize() 也是如此了，看了源码之后就明白，判断工作线程是否大于最大线程，如果大于，则对空闲线程发起中断请求。

    经过对两个方法进行分析，我们知道了最大线程数和核心线程数是可以动态调整的。

    # 动态设置的注意点是哪些？

    可以看看这个修改：

    代码图片 (opens new window)

    核心线程数修改了，但是最大线程数不动；我们此时再来测试看看结果：

    代码图片 (opens new window)

    结果发现我们的核心线程数虽然是修改了，但是活动的线程数还是原始值的最大线程数，没有变化呀！怎么办呢？

    我们可以看看 ThreadPoolExecutor#getTask() ，getTask() 里面有一段代码这样写着：即当工作线程大于最大线程数的时候，会对工作线程进行减 1 操作，然后返回 null；

    代码图片 (opens new window)

    所以这个地方的流程是：

    1. 创建新的工作线程 worker，然后工作线程数加 1 操作；
    2. 运行创建的工作线程 worker，开始获取任务 Task；
    3. 工作线程数大于最大线程数，对工作线程进行减 1 操作；
    4. 返回 null ，即没有获取到 Task；
    5. 清理该任务，流程结束。

    这样一加一减，所以真正在执行任务的工作线程数量一直没有发生变化，也就是最大线程数！

    如何解决这个问题呢？

    将我们的核心线程数设置了之后也同时需要设置我们的最大线程数即可！其实可以将二者设置为一样的。

    代码图片 (opens new window)

    而且不需要担心活动线程数调整得太大了，因为 setCorePoolSize() 方法提到，当 allowCoreThreadTimeOut 参数设置为 true 的时候，核心线程在空闲了 keepAliveTime 的时间后也会被回收的，相当于线程池自动给你动态修改了。

    # 如何动态指定队列长度？

    我们的线程池中没有对应的改变队列的容量的方法，因为我们的阻塞队列的 capacity 是 final 修饰的；所以我们需要自己自定义一个阻塞队列，将我们的 capacity 该用 volatile 修饰即可；然后就达到了动态修饰阻塞队列的目的啦！
