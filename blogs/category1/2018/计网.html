    • get 和 post 请求的区别

      • get 请求的参数数据一般都是在 url 中展示的，post 请求的数据一般是以 json（任意形式） 形式通过TCP协议进行发送过去的；

      • 安全 和 幂等：在 HTTP 协议中，所谓安全的方法就是不会破坏掉服务器上的资源；幂等就是 多次执行相同的操作，返回的结果是相同的。

        从这方面说，get请求是从服务器中获取资源，并不会改变服务器的资源，那么可以说是安全 且 幂等的；

        但是post请求的话，他是会改变服务器的资源的，每次post之后都会创建新的资源，所以post不是幂等也不是安全的；（但这只是一般的情况下）

    • HTTPS 解决了 HTTP 的啥问题，怎么解决的？

      混合加密 解决了HTTP中信息被被窃听的危险；

      摘要算法 + 数字签名 解决了HTTP中信息被篡改的危险；

      数字证书 解决了HTTP中信息被冒充的危险。

      混合加密：通信前，使用的是非对称加密手段（两把密钥进行操作）；通信后，使用对称加密的手段进行信息传输

      我们担心传输的信息被篡改被冒充，但是这些都不是事；【私钥加密，公钥解密】 这是一个可以防止信息被冒充的一个办法，因为私钥是独有的不可泄漏的，公钥是给每个人进行颁发的，如果我这个利用公钥能解开你的私钥，那么这个信息就只能是你的了，解决了冒充信息的缺点。

      图片展示 (opens new window)

      私钥是服务端进行保管的，然后服务端向客户端颁发对应的公钥。客户端收到的消息如果可以被公钥进行解密，那么就说明这个消息是由服务器发送的。

      但是问题来了，这个颁发后的公钥有可能是伪造的呀~那样的话还是会出现信息篡改的问题。

      服务器先将自己的公钥去到CA进行申请数字证书，CA用自己的私钥将 服务器发来的 公钥 + 数字证书进行加密，然后CA自己的公钥是存储在浏览器或者是OS中的。客户端是会拿到这个CA的公钥，然后对数字证书进行解密，获得 服务器的公钥 + 数字签名，然后对信息进行信息加密后传输，然后服务器再根据自己的私钥进行解密的。以上这个流程，通过 数字证书的方式来保证了服务器公钥的身份，避免了被冒充的风险。

    • HTTPS是如何建立连接的，期间的交互是如何的？

      HTTPS建立连接，相对于HTTP连接，就是多了一层 TLS/SSL这个基本的连接（这里只需要讲基于EDCHE算法实现的即可）

      • 客户端发送【clienthello】，其中包含的信息为 TLS 版本；随机数；密码套件；

      • 服务端发送【server hello 】消息 确认客户端发来的TLS版本，然后自己也生成一个随机数，然后根据密码套件列表中选择一个合适的密码套件；为了证明自己的身份，还会发送【certificate】将 数字证书 发送给客户端；随后还会发送【Server Key Exchange】消息给客户端，这个过程干了三件事：

        • 选择一个椭圆曲线，相当于在选好了一个椭圆曲线的基点G，会公开给客户端；
        • 生成一个随机数作为服务端的私钥，保留在本地；
        • 根据基点G和私钥计算出 服务端的椭圆曲线的公钥（这个公钥会进行RSA算法进行数字签名 存储在 数字证书上），公开给客户端；

        随后就发送【server hello down】消息，服务端给客户端表明：我消息已发完；

      • 客户端会根据在浏览器或者OS的CA公钥来进行对服务端发来的数字证书进行校验，会得到服务器的公钥，无误后往下走；

        客户端也会自己生成一个随机数作为自己椭圆曲线的私钥，然后并且根据服务端前面发来的消息生成客户端的公钥，然后用【client Key Exchange】消息发送给服务端；

        此时 就可以算出 x 的密钥条件之一：利用自己椭圆曲线的私钥 + 对方椭圆曲线的公钥 + 椭圆曲线基点G算出；

        真正的会话密钥：客户端随机数 + 服务端随机数 + x；

        然后客户端最后再发送这个【Change Cipher Spec】消息告诉服务端后续改用这个对称算法加密通信（并且 这里可以携带真正的数据过去的）；客户端还需要发送一个消息过去【Encrypted handshake Message】，将之前的消息做一个摘要然后加密发送过去给服务端验证本次的密钥是否正常使用；

      • 服务端也会回应一个【Change Cipher Exchange】和【Encrypted handshake Message】也做一个改变通信方式以及验证加密是否可用，就建立完连接了；

    • 301 和 302、304 状态码的区别

      301 表示永久重定向；需要用新的URL才能进行访问；

      302 表示临时重定向；需要暂时用另一个URL才能进行访问；

      304 不具有跳转的含义，表示资源未修改，重定向已经存在的缓冲文件，也称缓存重定向，也就是告诉客户端可以继续使用本地磁盘缓存的资源。

    • 强制缓存 以及 协商缓存

      • 强制缓存：只要浏览器判断缓存没有过期，直接使用浏览器的本地缓存，决定是否使用缓存的主动性在于浏览器这边；

      • 协商缓存：客户端与服务端协商之后，通过协商结果来判断是否使用本地缓存；

    • HTTP 1.1、HTTP 2.0、HTTP3.0都有啥特点

      • HTTP 1.1 相对于 HHTP1.0 来说，改变了两点：第一、变成长连接了；第二、变成管道运输，支持多个请求发送，不需要等到响应回来再去发送；

      • HTTP 2.0 是基于 HTTPS的；

        第一个特点：头部压缩，减少这个空间；HPACK 算法：在服务端和客户端同时维护了一张头信息表，所有字段都会存入这个表中，同时发送多个请求，请求头是一样的话，协议会自动消除重复的部分；

        第二个特点：由以前的文本传输，变成二进制传输，空间减少；头信息的报文变成 头信息帧；数据体变成数据帧

        图片展示 (opens new window)

        第三个特点：支持并发传输；一个 TCP 链上有多个 Stream，而一个Stream上有多个Message，所谓的 Message 就是我们的请求-响应模型，一个Message上有多个frame，我们的frame就是我们的消息，即数据帧（头部和包体），可以乱序发送的，接受的时候会自己进行组装信息；（但是很遗憾没有办法解决这个TCP层面队头阻塞问题：因为多个Stream在一条TCP链上的，只要有一个数据帧丢失了，这个TCP链都需要被阻塞，直到这个丢失的数据帧重新发回）

        图片展示 (opens new window)

        第四个特点：服务器可以主动推送消息给客户端；当客户端发送一次请求过去给服务端的时候，服务端可以一次性将html 以及 css 页面返回；

      • HTTP3.0 是基于UDP的QUIC协议进行传输的，可以完美解决这个队头阻塞问题；

        第一个特点：HTTP3.0 是无队头阻塞的，当某个流发生丢包问题的时候，只会阻塞这个流，并不会阻塞其他流，因此存在这个队头阻塞的问题；为什么可以这样？QUIC也是借鉴了HTTP2.0的一个QUIC链上多个Stream流，但是QUIC协议给每个stream流安排了一个单独的滑动窗口；

        第二个特点：更快的连接建立，我们知道HTTP2是基于HTTPS的，需要经过SSL的四次握手以及TCP的三次握手，耗时 3-RTT 时延；但是HTTP3.0的基于UDP的QUIC协议是很快的，其封装了SSL1.3+版本的，只需要一个RTT时间即可；

        第三个特点：连接迁移更快，我们知道TCP是根据目标IP，目标端口号，主机IP，主机端口号进行连接的；所以每一次的切换网络的时候都是比较耗性能的，但是这只需要双方建立连接 ID 即可进行重连。

    应用层：

    • 主要的协议：HTTP、FTP、DNS

      这个层主要是用户能真正接触到一个层，专注于给用户提供应用功能；该层位于用户态；其余的都是在内核态

    传输层：

    • TCP、UDP

      该层负责对数据进行传输；当数据从HTTP传来时，对数据进行组装；该层传输的数据包含了端口号；

    网络层：

    • IP

      该层负责的是网络的连接，将数据从一个设备传输到另一个设备；其中网络连接需要知道网络号和主机号，如何求得这两个号就需要IP的数值和子网掩码进行按位运算取得的；

      还有一个功能就是路由：通过路由算法来找到下一个路径在哪，路由器寻址工作中，就是要找到目标地址的子网，找到后进而把数据包转发给对应的网络；

    网络接口层：

    • MAC

      该层协议主要是将数据封装成帧进行传输了；利用ARP协议获取对方的MAC地址即可；

    • [ ] TCP HTTP的粘包拆包问题，有哪些方案可以解决

    • 如何解决粘包拆包问题？

      • TCP是基于流的传输，没有明显的界定条件；
      • 应用程序读取的速度不匹配：当接收方读取数据的速度和发送方发送数据的速度不一致的时候，无法判断数据包的边界也会导致粘包问题；
      • 发送方将数据写入缓冲区的时候是等到数据量较大的时候才进行发送的，所以导致多个小数据包被合并导致粘包拆包问题；

      TCP中有三个方案：

      • 固定的数组进行装载，但是缺点也很明显：就是如果数据量很小的时候就会导致空间浪费；
      • 使用特殊的符号，比如HTTP使用空格进行分割，或者可以用 “\r\n”；
      • 使用头部 + 数据包的方式：头部有一个固定的字段指定了后面数据包的大小；

      HTTP中如何解决的？

      HTTP不会出现粘包半包问题，在长持久的连接中可能会出现多个信息混在一起的情况，HTTP1.1中是利用了空行，每个消息之间都有一个空行来进行分割的；HTTP2.0中是利用了帧头信息进行分割，其中帧头就是我们第三种方案嘛

    • 紧急指针

      TCP是面向数据流的协议，当应用程序发生紧急情况（比如说被中断连接的时候），要求在接收方没有处理完数据之前就能够发送一些紧急数据，然后发送方会将urg字段标为1，表示为紧急指针有效，这样就不需要考虑你数据在数据流的位置，直接升级为优先级最高。

    • 流量控制

      TCP提供的一种让【发送方】根据【接收方】的实际接收能力来控制数据的传输；

    • 滑动窗口

      窗口的大小是 不需要进行响应，而可以继续发送的最大数据量；

      滑动窗口是【接收方】告诉【发送方】自己的实际接受能力，不至于发送方发送过多的数据让接收方没办法处理。

    • 拥塞机制

      • 几个比较重要的名词：慢启动，慢启动门阀，拥塞窗口、轮次
      • 四个算法：慢启动（指数相加）、拥塞避免（线性相加）、拥塞发生、快速恢复

    # 其他

    • [ ] 输入一个 url ，会发生啥？（不熟悉）

      涉及到三个进程：浏览器进程、网络进程、服务器

      浏览器负责检查和组装我们的协议，构成完整的 url；然后浏览器通过进程间的通信将我们的 url 请求交给我们的网络进程；

      网络进程收到我们的 url 后进行检查本地缓存是否缓存了该请求内容，如果有则返回，没有则向 web 服务器发送 HTTP 请求，请求流程如下：

      • 如果是域名的话就进行 DNS 解析，获取我们的 IP 地址和端口号；
      • 利用 ip 和 端口号和服务器进行 tcp 连接
      • 构建请求头并且发送
      • 服务器响应后，网络进程接收请求头和响应信息，并且解析响应内容

      网络进程解析响应内容流程：

      • 如果状态码是 301（永久重定向） 302（临时重定向），就需要重定向；如果是 304 就缓存重定向即可
      • 如果是 200 就代表响应内容获取成功，如果是字节流就使用下载管理器进行处理，如果是 html 类型的就进行渲染（渲染进程处理）操作即可

    • [ ] cookie session token 是啥，作用是啥

    • cookie session token 是啥，作用是啥

      • cookie一般是用来存储用户相对不那么重要的信息的，因为http是无状态连接嘛 哈哈 ；服务器产生的，存储在客户端；
      • session 一般是用来存储用户的认证那些比较重要的消息的，服务端产生，存储在客户端；
      • 因为我们的app 公众号 小程序没有浏览器嘛，就需要token，用来做身份鉴权的，限制某些资源的访问，服务端产生，服务端存储；

    补充一下：cookie 是保存在客户端的一小块文本串的数据，可以根据这个 cookie 判断用户的身份和状态； session 指的就是服务端和客户端一次会话的过程 ，用于记录客户状态的机制；

    作用：

    • 存储时间来看：cookie 长期可以，但是 session 是短期的；
    • 存储大小不同：cookie 不超过 4k，但是 session 存储的数据远远大于 cookie的；
    • 还可以利用 cookie 记录 session 的标识，第一次访问服务器的时候会创建一个 session 进行存储在我们的服务器中，并且返回 sessionId 给我们的 客户端中的 cookie 进行存储，第二次进行访问的时候会根据这个 sessionId 来判断我们的登录是否登录或者是登录失效，如果找到 Session 证明用户已经登录可以执行后面的操作了；

    # HTTP

    • [ ] HTTP 和 HTTPS 的区别？

      端口号：80 443

      保密性：S 多了个 SSL/TLS 中间层

      HTTPS 通过向 CA 申请数字证书解决了 HTTP 的被冒充的可能，同时也避免了被篡改的可能

      这里可以说如何解决这个被冒充的；如何解决信息不被篡改的等。

    • [ ] HTTPS 的连接过程？

      四次握手：

      • 第一次握手：

        客户端：客户端发送请求连接给服务端，请求的端口为 443；与此同此还会发送客户端随机生成的随机数以及SSL版本号以及密码套件给服务端；

      • 第二次握手：

      服务端：服务端接收到客户端的请求后，检查这个 SSL 的版本号确定没问题了，就发送 CA颁发的数字证书 给客户端，其中这个证书包含着公钥（这个公钥是利用 服务端自己生成的私钥 + 椭圆曲线的G点 （是基于 ECDHE 这个算法不是 RSA 这个算法）生成的）以及证书的失效时间等信息；

      • 第三次握手：

        客户端：客户端利用 浏览器缓存的 CA公钥 验证这个数字证书，没问题之后，获取到服务端的公钥，同时客户端自己也会生成一个随机数来作为椭圆曲线的私钥，所以就生成一个随机码数（自己的私钥 + 对方的公钥 + 椭圆曲线G + 客户端的随机数 + 服务端的随机数）作为会话密钥，并且使用 服务端的公钥进行加密，然后传送给服务端；

      • 第四次握手

        服务端：服务端利用这个 先前保留的 自己的私钥进行解密，获取到了随机码后，使用随机码进行加密信息，客户端收到信息后也会使用这个随机码解密信息，从此加密通信形成了。

      图片 (opens new window)

    • [ ] 客户端如何校验 CA证书 的合法性？

      • CA签发证书的过程：

        CA 机构会将持有者的公钥、用途、有效信息等打成一个包，然后对这些信息进行 Hash 运算；

        然后 CA 机构利用这个私钥对这个 Hash 值进行一个加密处理，生成一个证书签名，也就是 CA 对证书做了一个数字签名；

        最后将这个数字签名添加在文件证书上，形成数字证书；

      • 客户端校验服务端的数字证书的过程

        客户端会利用同样的 Hash 函数进行获取到我们的 Hash 值 H1；

        然后客户端利用 浏览器中的 CA的公钥信息进行解密证书中的数字签名，获取到我们的 Hash 值 H2；

        最后比较 H1 和 H2，如果值相同，则为可信赖的证书。

      假如中间有人篡改了证书的内容，他又没有 CA 的私钥，所以说，客户端用公钥进行解密的内容就不一致了。

    • [ ] 为什么需要 2msl？

      这个问题知道之前，首先要知道为什么要 TIME_WAIT 阶段？

      只有主动关闭连接的一方，才有这个 TIME_WAIT 阶段；

      1. 防止历史连接中，被后面相同的四元组 (opens new window)的连接错误的接收；
      2. 保证【被动关闭连接】的一方，能被正确地关闭；TIME-WAIT 作用是等待足够的时间以确保最后的 ACK 能让被动关闭方接收，从而帮助其正常关闭。

      好了，了解了为什么需要 TIME_WAIT 之后，我们来看看为什么需要 2msl，就是：

      我们允许报文丢失一次；如果被动关闭方没有接收到断开连接的最后的 ACK 报文，就会触发超时重发 FIN 报文，另一方接收到 FIN 后，就会重发 ACK 给被动关闭方，一来一去正好 2 个 msl，而且我们需要知道，2MSL 的时间是从客户端收到 FIN 后发送 ACK 开始计时的；

    • [ ] 粘包和半包问题如何产生？又如何解决？

      • TCP是基于流的传输，没有明显的界定条件；
      • 应用程序读取的速度不匹配：当接收方读取数据的速度和发送方发送数据的速度不一致的时候，无法判断数据包的边界也会导致粘包问题；
      • 发送方将数据写入缓冲区的时候是等到数据量较大的时候才进行发送的，所以导致多个小数据包被合并导致粘包拆包问题；

      TCP中有三个方案：

      • 固定的数组进行装载，但是缺点也很明显：就是如果数据量很小的时候就会导致空间浪费；
      • 使用特殊的符号，比如HTTP使用空格进行分割，或者可以用 “\r\n”；
      • 使用头部 + 数据包的方式：头部有一个固定的字段指定了后面数据包的大小；

      HTTP中如何解决的？

      HTTP不会出现粘包半包问题，在长持久的连接中可能会出现多个信息混在一起的情况，HTTP1.1中是利用了空行，每个消息之间都有一个空行来进行分割的；HTTP2.0中是利用了帧头信息进行分割，其中帧头就是我们第三种方案嘛

    # IP

    这里写一些 IP 层的基础知识；

    • 什么是 IP 地址？作用是啥？

      IP 地址分为 IPV4 和 IPV6，IPV4的表示的方式是通过【.】来进行分割的，每一个代表十进制；共 32 位，其中分为 4 组，每组 8 位；IPV6的表示的方式是通过【:】进行分割，共 128 位，每 16 位为一组；

      这里又会延申一个重要的问题：IPV4 和 IPV6 有哪些区别？

      除了上面的标识不一样外，还有 IPV6 能给每个计算机都分配一个公有的 IP 地址，这样的话就不需要 NAT 技术了；同时还去掉了包头校验和，简化首部结构，减轻了路由器负荷，大大提升了传输性能；

      作用就是 将数据包从一个主机经过网络上传送到另一个主机；

    • IP 分片和重组

      IP 数据包大于 MTU 时，IP 数据包会被分片，经分片之后的 IP 数据在进行重组的时候，只能由目标主机进行，路由器是不会进行重组的；

      在分片传输过程中，一旦某个分片丢失了，会造成整个 IP 数据报作废，所以 TCP 引入了 MSS 也就是在 TCP 层进行分片不由 IP 层分片，那么对于 UDP 我们尽量不要发送一个大于 MTU 的数据报文；
