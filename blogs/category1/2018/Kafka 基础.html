    考小拉 10/1/2023 Kafka秋招

    # Kafka 架构

    Kafka 的队列模式是发布-订阅模式，使用 主题（TOPIC）作为消息的通信载体，类似于广播模式，发布者发布一条消息，该消息通过主题传递给所有的订阅者，在一条消息广播之后才订阅的用户是是收不到这个消息的；

    RocketMQ 的消息模型和 Kafka 基本是完全一样的。唯一的区别是 Kafka 中没有队列这个概念，与之对应的是 Partition（分区）

    # Kafka 的消费顺序消息丢失和消费重复

    • 消费顺序

      比如说有一个场景：比如说我们同时发了 2 个消息，这两个消息对应的操作分别对应的数据库操作是：

      1. 更改用户会员等级
      2. 根据会员等级计算订单价格

      上面的这两个消息是不能乱序的。。。后果严重

      Kafka 中的 Partition（分区）才是真正保存消息的地方，发送的消息放在这里，而我们的 Partition（分区）又存在于 Topic（主题）这个概念中；并且我们可以给特定的 Topic 指定多个 Partition。

      Kafka 只能保证 Partition（分区）中消息有序，消息被追加到 Partition（分区）的时候都会分配一个特定的偏移量（offset），Kafka 通过偏移量（offset）来保证消息在 Partition 的顺序性；

      基于上述说的消息顺序问题，我们业务中如何实现？

      1. 发送消息的时候指定 key/partition；将消费者划分为每组只有一个消费者，这样每个消费者只消费主机所属分区的消息，从而保证了每个分区内的消息消费顺序；只保证分区内的顺序，无法保证整个主题的消费顺序。
      2. 在消息中添加一个时间戳字段，消费者在处理消息的是可以根据时间戳进行排序；通过时间戳可以保证消费的顺序，但是需要注意同一时间戳会存在并发消费的问题。

    • 如何保证消息不丢失

      可以自定义设置回调函数进行显示成功还是失败

       ListenableFuture<SendResult<String, Object>> future = kafkaProducer.sendLotteryInvoice(invoiceVO);
        future.addCallback(new ListenableFutureCallback<SendResult<String, Object>>() {
            @Override
            public void onSuccess(SendResult<String, Object> stringObjectSendResult) {
                // 4.1 MQ 消息发送完成，更新数据库表 user_strategy_export.mq_state = 1
                activityPartake.updateInvoiceMqState(invoiceVO.getuId(), invoiceVO.getOrderId(), Constants.MQState.COMPLETE.getCode());
            }

            @Override
            public void onFailure(Throwable throwable) {
                // 4.2 MQ 消息发送失败，更新数据库表 user_strategy_export.mq_state = 2 【等待定时任务扫码补偿MQ消息】
                activityPartake.updateInvoiceMqState(invoiceVO.getuId(), invoiceVO.getOrderId(), Constants.MQState.FAIL.getCode());
            }
        });

    • 如何保证消息不重复消费

      Kafka 出现消息重复消费的原因：服务端侧已经消费的数据没有成功提交 offset （根本原因）

      解决方案：

      • 将消息服务做幂等校验，比如 Redis 的 set、MySQL 的主键等；
      • 手动提交 offset：但是问题来了，什么时候提交呢？拉取到消息即提交：会有消息丢失的风险。允许消息延时的场景，可以使用这个方案，然后通过定时任务，在业务不繁忙的时候做数据兜底。
