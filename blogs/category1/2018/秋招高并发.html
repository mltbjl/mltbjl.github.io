    # 锁

    乐观锁：当前线程认为自己使用数据时不会有别的线程进行修改，所以不需要添加锁；只是在更新数据的时候会先判断主存中的数据是否被其他线程修改过，如果没修改过则直接写入；否则重新检查数据；Java 实现乐观锁的方式为使用了 CAS + 自旋操作；同时 Java 中的原子类操作就是根据这个 CAS 自旋实现的；

    悲观锁：当前线程在进行使用数据的时候认为一定有其他线程进行数据的修改，因此在获取数据的时候会先加锁，确保数据不会被其他线程修改；Java 典型的操作就是 synchronized 以及 ReentranLock

    图片 (opens new window)

    从上面的描述可以知道

    • 悲观锁适合写操作多的场景，先加锁可以保证写数据的时候正确；
    • 乐观锁适合读操作多的场景，不加锁的特点能够使其操作的性能大大提升；

    乐观锁的实现方式：CAS

    CAS 即为比较并交换，当需要操作数据时需要先拿到主存中的数据进行比较，如果没有其他线程进行修改，就更新自己的数据；否则进行报错或者自旋；CAS 算法的三个变量：①需要读写的内存值 ②进行比较的值 ③需要写入的新值

    CAS 会导致三个问题：

    图片 (opens new window)

    • ABA 问题：就是说 我们对一个变量进行 CAS 操作的时候，我们以为这个值没有发生变化，但是其实他已经变化了，比如说，值原来是5，后面改成了10，在后面的时候改为了5这样；解决方法有两个：
      • 第一个是才去版本号，进行一次数据操作的时候版本号进行 +1，然后 原来的 A -> B -> C，变成了 1A -> 2B -> 3C；
      • 第二种可以使用这个 AtomicStampedReference 这个类，他有一个 compareAndSet 方法首先检查当前对象的引用值是否等于预期引用，并且当前印戳（Stamp）标志是否等于预期标志，如果完全相等，以原子的方式将引用值和印戳标志的值更新给指定的更新值了。
    • 自旋时间开销大：如果一个线程长时间获取不到锁，就有可能给 CPU 带来非常大的开销；
    • 只能保证一个共享变量的原子操作：对一个共享变量的时候，CAS 是可以保证这个原子操作，但是对于多个共享变量来说，CAS 就没办法保证；解决办法为：
      • 使用 AtomicReference 类来保证引用对象之间的原子性，可以把多个变量放在一个对象里来进行 CAS 操作

    公平锁和非公平锁

    公平锁：多个线程按照申请锁的顺序来获取锁，线程直接进入队列中排队，队列中第一个线程才能获取锁；

    • 优点：每个线程都不会被饿死；
    • 缺点：整个系统的吞吐量低，性能不那么好；

    非公平锁：多个线程加锁时直接尝试获取锁，获取不到就去等待队列的队尾进行排队等待；但如果此时锁刚好可用，那么这个线程就会可以无需阻塞直接获取锁，所以非公平锁可能会出现后申请锁的线程获取到锁的情况；

    自旋锁：

    在了解自旋锁之前，首先我们先要知道，为什么需要自旋锁？

    阻塞和唤醒一个线程，是需要 OS 进行 CPU 上下文切换的，而这个切换是需要消耗处理器的时间，但是一般来说，锁住的代码块执行时间很短，速度快；所以会出现 执行代码所需要的时间 < CPU上下文切换所需要的时间，本末倒置了；所以基于这个问题，比如说当两个线程进行抢夺资源（锁）时，如果另一个线程没抢到，就先暂时不要挂起，而是继续使用 CPU 的执行时间，来看看持有锁的线程是否很快地释放锁；这就是自旋锁的由来；在我们的并发包中，Atomic 原子类就是通过 CAS + 自旋 的操作进行保证原子性的。

    图片 (opens new window)

    但是这个自旋锁也有缺点的：自旋操作虽然避免了线程切换的开销，但是要占用处理时间的；如果被占用的时间很短，自旋等待效果很好，但是很长的时候会浪费处理器（CPU）的资源，所以自旋到一定次数还在等待，那就要挂起这个线程了。

    图片 (opens new window)

    独占锁和共享锁：ReentranLock 和 ReentranReadWriteLock

    重点来说这个 ReentranReadWriteLock 这个锁，顾名思义：读锁 ReadLock 则为共享锁；写锁 WriteLock 就是独占锁，也是排他锁；底层都是 Sync 这个类来实现的；我们一般用读锁来进行 并发读的操作，并发写的时候只能用 独占锁；读读是可以并发的，但是读写，写读，写写都是互斥的；

    也可以看看这个关于 Java 锁的文章 (opens new window) 主要是写一些示例 demo

    在看我们的读写锁如何实现之前，我们先了解我们的共享变量 state ，它是一个 int 整型的 32位的变量，在独占锁中，值一般位0或者1（如果是重入锁的话就是重入的次数）；在共享锁中，代表的就是持有锁的个数；但是在读写锁中，就将 state 变量分为高16位和低16位两个维度；高16位表示的是读锁个数，低16位表示写锁个数。

    图片 (opens new window)

    先来看看写锁如何实现的：在 tryAcquire() 方法中，判断当前是否有读锁以及我们的当前占据锁的线程是否为当前线程；如果是则返回成功否则返回失败；

    再来看看读锁如何实现的：在 tryAcquireShare() 方法中，不说那么多，只需要知道如果当前锁是独占锁，则意味着有线程获取了写锁，需要等待；

    轮到我们的 ReentranLock，不论是公平锁还是非公平锁，都是独占锁；因为他都会进行判断：当前线程是否为拥有锁的线程，是才进行 state 变量的修改，否则返回 false

    Synchronized 锁升级策略：无锁 --> 偏向锁 --> 轻量级锁 --> 重量级锁

    主要是跟 对象内存中的对象头有关系，然后还跟我们的 Minitor 有关系

    无锁开始，所有的线程都可以访问共享资源，但同时只有一个线程可以修改成功；无锁的特点就是在一个循环内不断尝试修改这个共享资源，CAS 应用就是无锁策略；但是无锁没办法代替有锁，但无锁在某些场景性能还是很高的；

    偏向锁，即该对象锁总是被一个线程占有，该对象头里记录了该线程的 ThreadId ；在线程进入和退出同步块时不再通过 CAS 操作进行加锁和解锁，而是检测 Mark Word 里是否含有存储着指向当前线程的偏向锁。引入偏向锁的目的是为了减少不必要的轻量级执行路径，因为轻量级锁的获取以及释放依赖多次的 CAS 原子指令；而偏向锁只需要在置换 ThreadId 的时候依赖一次 CAS 原子指令操作；偏向锁只有遇到竞争线程的时候，持有偏向锁的线程才会释放锁，线程不会主动释放偏向锁，而是等到一个全局的安全点（就是没有字节码运行的时候），它会暂停拥有偏向锁的线程，判断锁对象是否还是处于锁定状态，撤销锁后恢复到无锁或者轻量级锁状态。

    轻量级锁，当对象锁是偏向锁时，如果有其他线程来竞争，偏向锁就会升级为轻量级锁，其他线程不会被阻塞，而是进行自旋尝试获取锁，从而提高性能，问题是偏向锁如何升级到轻量级锁呢？我们的虚拟机会给我们当前线程栈帧（存在我们的栈中）中生成一个锁空间（Lock Record），然后将对象的 Mark Word 拷贝到我们栈帧中的锁记录中，成功之后，尝试使用 CAS 操作将对象的 Mark Word 更新为指向 Lock Record 的指针，并将锁记录中的 Owner 指针指向对象的 Mark Word；这个动作成功之后，Mark Word 的锁标记也需要改变为 00，表示此时处于轻量级锁状态；如果是失败的话，虚拟机会首先检查对象的 Mark Word 是否还是指向我们当前的线程，如果是则代表当前线程持有该轻量级锁，则继续进行同步块操作，否则就说明出现竞争锁；

    当一个线程持有锁，另一个线程不断地自旋，当自旋到一定次数的时候，或者说，有第三个线程参与竞争的时候，轻量级锁就升级为重量级锁；

    升级为重量级锁，Mark Word 的标记位就会变成 10，此时 Mark Word 存储的是指向重量级锁的指针，此时等待的线程就会进入阻塞状态。

    综上：

    偏向锁是通过对比 Mark Word 解决锁的问题，避免了 CAS 操作导致的多次的原子指令；轻量级锁则是通过 CAS + 自旋操作避免线程被挂起；重量级锁将除了拥有锁的其他线程挂起；

    # volatile

    • 了解 volatile 之前需要了解 Java 的 JMM 即 Java 内存模型、

      图片 (opens new window)

      这个图片是一个双核 CPU 系统架构，每个核有自己的控制器和运算器，控制器包含一组寄存器和操作控制器， 运行器执行算数运算逻辑，每个核都有自己的一级缓存，有些架构里面还有一个所有 CPU 共享的二级缓存。Java 内存模型中的工作内存，对应就是一级缓存或者是二级缓存。

      JMM 定义了线程和主内存的关系：线程之间的共享变量存储在我们的主内存中，每个线程都有自己的一个独立的本地内存，本地内存存储了该线程以读\写共享变量的副本（线程操作的地方就是本地内存嘛）。

    • JMM 的核心三大特性：原子性；可见性；有序性

      原子性：

      可见性：一个线程修改了某一个共享变量的值时，其他线程会立刻知道这个修改

      有序性：有序性指的是一个线程执行代码，从前往后依次执行

      如何保证呢？

      原子性可以通过 sychronized 来进行保证，可见性是通过 volatile 或者 final 或者 synchronized 来进行保证，有序性通过 synchronized 或者 volatile 来进行保证的。

    • 什么是指令重排？

      线程在执行指令的时候，为了提高性能，编译器和处理器会对指令进行重排序；

    IO 密集型 和 CPU 密集型如何选择计算：

    CPU密集型：cpu + 1；IO密集型：cpu * 2

    为什么是这样计算呢？

    CPU 密集型是由于计算任务占据主要的 CPU 资源，我们通常将 CPU 数量 + 1来进行充分使用多核处理能力，这样做的目的是避免 CPU 资源饱和，保持任务的高效运行；

    IO 密集型主要的任务消耗在我们的 IO 操作上，CPU 的利用率相对很低，这种情况下，为了更好利用 CPU 资源，可以提高这个 CPU 数来提高任务的并发处理能力；目的是在等待 IO 操作完成的过程中，可以并行处理其他任务，从而减少等待时间，提高系统的吞吐量。

    # Synchronized 和 ReentranLock 对比

    图片 (opens new window)

    区别：

    • 一个是 JVM 层面的锁，一个是 API 层面的锁；
    • Synchronized 锁底层是基于 Monitor 对象锁进行实现的，这个 Monitor 是底层是基于 Metux Lock 互斥锁实现的；ReentranLock基于这个 AQS 框架实现的；
    • Synchronized 一般可以作用于方法上，静态方法上，以及代码块；ReentranLock 一般都是作用在代码块上的，然后需要手动释放锁，而且也可以设置一个锁的过期时间；
    • 两者都是非公平锁（ReentranLock 默认是）；都是可重入锁；
    • synchronized 不够灵活，ReentranLock 灵活，提供了锁中断、锁精准通知、锁超时等功能。

    ConcurrentHashMap 的 put() 执行流程、get() 执行流程；key 和 value 为什么不可以是 null？

    • put() 流程：首先根据 key 计算哈希值 ---- 》然后循环遍历桶数组，如果当前数组为空，则进行初始化（CAS 操作进行扩容）----》 如果当前索引指向的位置为空，则进行 CAS 操作插入节点（无需加锁的桶数组）----》 如果当前桶被标记为迁移状态（MOVED），则帮助进行迁移 ----》 对当前 Node 节点进行锁定，确认并发情况下的线程安全 ----》 然后开始遍历节点，如果 key 相同则进行替换 value，如果遍历到尾部了还是没有对应的 key 就需要进行尾插法进行添加元素。

    • get() 流程：不需要加锁，但是前提桶数组不能为空；get() 方法是采用了 unsafe 类也就是 CAS 机制来保证获取到的元素是最新的；即采用了 volatile 来修饰 table 桶数组来保证集合的可见，来保证线程安全；
