    MySQL 的死锁请了解一下

    考小拉 10/20/2023 秋招MySQL

    该篇文章参考 小林 coding (opens new window)

    场景如下：有个业务的主要逻辑就是新增订单、修改订单、查询订单等操作；然后因为订单是不能重复的，所以当时在新增订单的时候就需要做一个幂等性校验，做法就是在新增订单的时候，先通过 select ... from ... where ... for update; 语句进行查询订单是否存在，如果不存在才插入订单记录。正是因为这样的操作，如果业务量很大的时候，就可能会出现死锁；

    # 死锁的发生

    我们先来模拟一下死锁：按照这个图来 (opens new window)

    但是我的 MySQL 是设置了死锁检测的哈哈，就没办法演示阻塞的状态了。。。

    问题来了，原理上为什么会出现这个死锁呢？

    在看这个文章前，我们先需要熟悉 MySQL 如何对数据进行加锁的 (opens new window) ；

    OK，看完文章过后，我们知道，我们进行执行 select ... for update 语句后，为了避免幻读的问题，我们事务 A 对 (1006, 无穷大] 这个区间加了临键锁，但是无穷大这个数不存在，其实也可以理解为间隙锁；然后我们的事务 B 也同样执行了 select ... for update 的语句后，对我们的 (1006, 无穷大] 这个区间加了临键锁，但是无穷大这个数不存在，其实也可以理解为间隙锁；

    间隙锁之间虽然也有 X 型锁和 S 型锁，但是是兼容的！

    完了之后，我们的事务 A 想往表中进行插入一条记录为 order_no = 1007 的记录会被阻塞；问题就来了：为什么会被阻塞呢？

    当我们执行插入语句的时候，会在插入间隙上获取插入意向锁，而插入意向锁与间隙锁之间是冲突的，所以当其他事务持有该间隙的间隙锁时，需要等待其他事务释放间隙锁之后，才能获取到插入意向锁；而间隙锁与间隙锁之间是兼容的，所以所以两个事务中 select ... for update 语句并不会相互影响。

    所以为什么会发生死锁的原因就是：事务 A 和 事务 B 在执行完 select ... for update 语句后都持有范围为(1006, 无穷大] 的临键锁，而接下来的插入操作是为了获取到插入意向锁，都在等待对方释放这个间隙锁，于是就造成了死锁了。

    补充一个知识：为什么间隙锁和间隙锁兼容呢？插入意向锁是什么？

    间隙锁的意义在于阻止区间被插入，因此是可以共存的。但是我们需要知道 临键锁 是包含着 间隙锁+记录锁的；我们如果是 X 型的临键锁，那么就不能共存了，因为还有我们的记录锁嘛~

    第二个问题：插入意向锁不是表锁那个意向锁，而是一个特殊的间隙锁，锁住的是一个点；

    插入意向锁是一种特殊的间隙锁，但不同于间隙锁的是，该锁只用于并发插入操作

    插入意向锁的生成时机：没插入一条记录，都需要看一下待插入记录的下一条记录是否加了间隙锁，如果已经加了间隙锁，此时会生成一个插入意向锁，然后锁的状态设置为 等待状态，只有锁的状态设置为 正常状态才算是该事务获取到该锁；
