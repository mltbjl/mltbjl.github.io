    Spring 的容器事件和监听机制

    考小拉 2/12/2024 Spring 的学习观察者模式设计模式

    我们可以查阅资料，知道 Spring 中有一个事件容器和事件监听器，是观察者模式的具体实现；

    我们学习一个东西的时候，按照这个思路：干什么 ---> 有什么作用 ---> 如何实现

    我们首先要知道这个事件容器和容器事件的监听机制作用是啥？

    # 为什么要有这个事件容器？监听机制是啥玩意？

    说人话：就是为了系统的解耦和模块化设计；

    容器事件的优点和应用场景：

    • 容器的初始化和销毁：容器事件可以在容器的初始化和销毁过程中进行监听；通过监听容器初始化事件，可以在容器初始化完成后执行一些对 Bean 初始化的操作；通过监听容器销毁的事件，可以在容器销毁前执行一些清理操作；
    • 容器内部的状态的监控：容器事件可以用于监控和统计容器的内部状态；通过监听容器事件，可以实时获取容器的状态的信息，比如 Bean 的加载数量、加载时间等操作；
    • 容器事件可以用于扩展容器的功能；通过监听容器事件，可以在容器的某个阶段做一些额外的处理，比如在 Bean 加载完成之后进行一些后置的操作，或者在容器初始化完成之后进行一些特定的操作；

    事件监听的作用和场景：

    • 解耦：通过事件的监听，模块就不直接依赖于其他模块，通过事件发布和监听来实现模块之间的通信；这样一来，只需要发布一个事件，其他监听该事件的模块就会做出相应的处理逻辑；
    • 扩展：当某个业务需要在特定的条件下触发某些动作时，可以将这个条件定义为一个事件，然后在监听该事件的地方实现响应的逻辑即可。比如：①一个用户注册完成后事件推送消息、②用户下单完成之后发送一个 MQ 消息、③用户支付完之后可以发货等等。。。
    • 日志记录：可以通过事件监听来实现日志记录；当某个关键事件发布后，可以发布一个事件，然后在监听该事件的地方将相关信息记录在日志中，方便后续的排查和分析。如：在事件的发布和处理过程中可以插入切面逻辑。

    既然知道了一些事件和事件监听的基本概念，我们前面说了，事件监听就是观察者模式的具体的实现。那什么是观察者模式？

    # 什么是观察者模式？

    观察者模式又叫做 发布订阅模式：

    首先我们需要理清楚一些概念先：发布者：也称之为被观察者 （publisher）；订阅者：也称之为观察者或者是监听者（Listener）；其中 我们的广播器作为事件从发布者传递到监听者的媒介（Multicaster）；还有我们的事件（Event），我们发布者发布的就是事件，监听者接收的也是事件。

      定义完 事件媒介接口 ApplicationEventMulticaster 后，定义其实现类 AbstractApplicationEventMulticaster ，在该类中，主要是实现了 添加删除监听者的逻辑外，还实现了 一个方法 supportEvent() ，该方法的主要作用是：通过获取监听器的泛型接口信息，并根据泛型接口获取实际的事件类型，然后判断传入的事件对象是否是该实际类型的子类或者实现类，从而确定监听器是否支持该事件类型。

      源码处 (opens new window)

      public abstract class AbstractApplicationEventMulticaster implements ApplicationEventMulticaster, BeanFactoryAware {
    public final Set<ApplicationListener<ApplicationEvent>> applicationListeners = new LinkedHashSet<>();
	  // 省略其他代码
    ......
    public Collection<ApplicationListener> getApplicationListeners(ApplicationEvent event) {
        LinkedList<ApplicationListener> allListeners = new LinkedList<>();
        for (ApplicationListener<ApplicationEvent> listener : applicationListeners) {
            // 判断监听者是否是监听该事件的
            if (supportEvent(listener, event)) {
                allListeners.add(listener);
            }
        }
        return allListeners;
    }

    private boolean supportEvent(ApplicationListener<ApplicationEvent> applicationListener, ApplicationEvent event) {
        Class<? extends ApplicationListener> listenerClass = applicationListener.getClass();

        // 按照 Cglib 或者 Simple 不同的实例化方式类型， 需要判断获取目标 class
        Class<?> targetClass = ClassUtils.isCglibProxyClass(listenerClass) ? listenerClass.getSuperclass() : listenerClass;
        Type genericInterface = targetClass.getGenericInterfaces()[0];
        Type actualTypeArgument = ((ParameterizedType) genericInterface).getActualTypeArguments()[0];
        String className = actualTypeArgument.getTypeName();
        Class<?> eventClassName;
        try {
            eventClassName = Class.forName(className);
        } catch (ClassNotFoundException e) {
            throw new BeansException("wrong event class name: " + className);
        }
        // 判定此 eventClassName对象所表示的类或者接口与指定的 event.getClass() 参数所表示的类或接口是否相同，或是否是其超类或超接口。
        // isAssignableFrom是用来判断子类和父类的关系的，或者接口的实现类和接口的关系的，默认所有的类的终极父类都是Object。如果A.isAssignableFrom(B)结果是true，证明B可以转换成为A,也就是A可以由B转换而来。
        return eventClassName.isAssignableFrom(event.getClass());
    }
}

    # 总结

    观察者模式的核心就是让我们的发布者以及我们的监听者进行解耦；我们可以自定义一个事件，该事件继承我们的 ApplicationContextEvent 这个抽象类，然后自己去实现自己的业务逻辑；然后我们还需要自定义一个监听者负责处理 事件发布后的逻辑，该监听者实现 ApplicationListener<E extends ApplicationEvent> 这个接口并且实现 里面的 onApplicationEvent(E event) 方法即可。

    具体到 Spring 是如何实现的上面已经说过啦，简单复述一下：

    Spring 的发布订阅模式有四个模块：事件 Event、事件广播器也是事件媒介 即 Multicaster、事件发布者 Publisher、事件监听者 ApplicationListener

    我们定义好事件后（当然 Spring 框架自己定义了容器刷新（ContextRefreshEvent）和关闭（ContextCloseEvent）的两个事件），也需要定义一个事件监听器；在我们 AbstractApplicationContext 这个抽象类的 refresh() 方法中，先进行初始化我们的 事件媒介 SimpleApplicationEventMulticaster 后，开始注册我们的监听器，监听器会被添加到我们的事件媒介中的 Set 集合中；以刷新容器事件为例，当我们执行这个事件的时候，我们会发布这个事件，我们的事件媒介 applicationEventMulticaster 会通过遍历获取到该刷新容器事件对应的监听器的类型，然后就 执行实现了监听器这个接口的 子类中重写 onApplicationEvent(E event) 方法就行啦~
